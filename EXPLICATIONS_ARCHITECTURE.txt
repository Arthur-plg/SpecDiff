======================================================
RÉSUMÉ DE L'ARCHITECTURE : BENCHMARK SPECDIFF
======================================================

Ce projet est conçu pour évaluer l'accélération apportée par le Speculative Diffusion Decoding (SpecDiff) par rapport à une génération Auto-Régressive (AR) standard.

------------------------------------------------------
1. engine.py (Le coeur de l'algorithme)
------------------------------------------------------
C'est le fichier le plus important. Il contient deux classes principales :

A) EngineMetrics
   - Rôle : Stocker et calculer précisément toutes les métriques de performance demandées.
   - Métriques :
     * TTFT (Time To First Token) : Latence avant le tout premier token émis.
     * ITL (Inter-Token Latency) : Temps moyen entre l'émission de deux tokens. Dans le cas de SpecDiff, ce temps est moyenné sur la taille du bloc de tokens acceptés d'un coup.
     * Throughput : Le débit global (total_tokens / total_time) en tokens par seconde.
     * Taux d'acceptation (α) : Le ratio entre les tokens du draft acceptés et le nombre de tokens du draft générés.

B) InferenceEngine
   - Rôle : Charger les modèles en mémoire et exécuter les boucles de génération.
   - Méthodes clés :
   
     > __init__ et _init_mdlm :
       Charge le modèle cible (GPT-2 XL en Float16) de manière classique.
       Charge le modèle draft (MDLM). S'il n'est pas présent, il clone le repo GitHub. Il utilise "Hydra" (un gestionnaire de configuration) pour initialiser l'architecture de MDLM, puis télécharge les poids depuis Hugging Face (hf_dit).
       
     > standard_autoregressive :
       La baseline. Génère les tokens un par un de manière séquentielle en appelant GPT-2 à chaque itération. C'est lent mais c'est la méthode de référence.
       
     > generate_draft_mdlm :
       C'est la première étape du SpecDiff. 
       Logique : MDLM est un modèle de diffusion. Pour générer un bloc de texte conditionnel au prompt :
       1. On crée un tenseur : [PROMPT] + [MASK] * gamma
       2. On applique une boucle "Reverse Diffusion" sur T étapes.
       3. À chaque étape, MDLM débruite les [MASK]. 
       4. CRITIQUE : À la fin de chaque étape, on force la partie [PROMPT] à reprendre ses valeurs exactes. Ainsi, MDLM est contraint de générer une suite logique au prompt.
       5. Retourne les `gamma` nouveaux tokens (le draft).
       
     > speculative_diffusion_decoding :
       C'est l'implémentation de l'Algorithme 1 (SpecDiff).
       Boucle principale (tant qu'on n'a pas généré max_new_tokens) :
       1. Demande à MDLM de générer un draft de taille `gamma` (via `generate_draft_mdlm`).
       2. Vérification parallèle : On donne au modèle cible (GPT-2) le prompt + les `gamma` tokens d'un seul coup. GPT-2 calcule les probabilités pour TOUTE la séquence en un seul passage (très rapide).
       3. Vérification gloutonne (Greedy Verification) : On parcourt le draft de gauche à droite. 
          - Si la prédiction de GPT-2 correspond au token de MDLM, on l'accepte et on passe au suivant.
          - Dès qu'il y a une erreur (Rejet), on ajoute la correction de GPT-2, on jette le reste du draft, et on recommence la boucle (nouveau draft).

------------------------------------------------------
2. app.py (Interface Graphique)
------------------------------------------------------
   - Rôle : C'est le tableau de bord Streamlit.
   - Fonctionnalités :
     - Deux modes : Un mode "Visualisation" pour analyser les fichiers CSV locaux sans charger les IA, et un mode "Exécution" pour lancer le code sur GPU.
     - Fournit les sliders pour modifier `gamma` (longueur du draft) et `T` (nombre d'étapes de diffusion de MDLM).
     - Appelle les méthodes de l'engine, puis génère un beau tableau comparatif des métriques et affiche le Speedup global.

------------------------------------------------------
3. run_benchmark.py (Exécution Headless)
------------------------------------------------------
   - Rôle : Un script Python simple conçu pour Google Colab.
   - Logique : Il exécute les deux méthodes (AR puis SpecDiff) l'une après l'autre sans interface graphique, imprime les résultats dans le terminal, et sauvegarde les logs. Cela permet d'exécuter de lourds calculs sur les serveurs Google sans se soucier du réseau ou de l'interface web.

------------------------------------------------------
4. benchmark_utils.py (Utilitaires de Logs)
------------------------------------------------------
   - Rôle : Fonction de sauvegarde (save_logs).
   - Logique : Prend les objets `EngineMetrics` remplis par `engine.py`, en extrait les données (Throughput, TTFT, etc.) et les écrit dans un fichier CSV (`specdiff_benchmark_logs.csv`), en le créant s'il n'existe pas, ou en y ajoutant de nouvelles lignes.
